# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

A Metroid-style 2D platformer built with Godot 4 using GDScript. Claude Code drives development via the **godot-mcp-pro** MCP server, which gives Claude live access to the Godot editor (scene tree, debug output, error logs) over a WebSocket on port 6505.

## Stack

- **Godot 4.4+** — game engine (install via `brew install --cask godot`)
- **GDScript** — scripting language (not C#)
- **godot-mcp-pro** — paid MCP plugin ($5, itch.io); enables Claude to launch the game, read errors, and inspect the scene tree
- **VS Code + godot-tools extension** — external script editor

## Running the game

The game must be run from inside the Godot editor (no CLI run command). The standard workflow:

1. Open Godot with the project — this auto-starts the WebSocket server on port 6505
2. Start Claude Code: `claude` in the project directory
3. Claude uses godot-mcp-pro MCP tools to create/edit scenes and scripts; Godot hot-reloads changes automatically

Verify MCP is connected:
```bash
claude mcp list   # should show godot-mcp-pro
```

## MCP server config

godot-mcp-pro is configured in `~/.claude/settings.json`:
```json
{
  "mcpServers": {
    "godot-mcp-pro": {
      "command": "node",
      "args": ["/path/to/godot-mcp-pro/server/dist/index.js"],
      "env": {}
    }
  }
}
```

## Architecture conventions

- **Scenes**: each game entity is its own `.tscn` scene (player, room, camera)
- **Scripts**: GDScript attached to scene root nodes; player movement uses `CharacterBody2D`
- **State machine**: enum + `match` in `_physics_process` — no external state machine plugin
- **Tilemaps**: rooms use `TileMapLayer` with collision; camera locks to room boundaries Metroid-style
- **Camera**: separate scene with follow logic, not a child of the player node

## Ignored paths

`.godot/`, `*.uid`, `export_presets.cfg` are auto-generated by Godot and not committed.

## Testing & QA

**Read `TESTING.md` before writing any QA/playtest sequence that drives the running game through MCP.** It documents how MCP async timing interacts with physics (short version: `simulate_action` followed by separate `capture_frames` calls will miss mid-motion states because of tool round-trip latency). TESTING.md has the three working patterns and pitfalls to avoid.
